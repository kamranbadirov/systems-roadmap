Personal Learning Roadmap

From CS Graduate to
Systems & PL Mastery

A structured, project-driven path through Computer Architecture, Operating Systems, Compilers, Programming Language Theory, and Runtimes — built for deep understanding, not surface familiarity.

6 Phases
18–24 Months
1–2 hrs Per Day
6 Projects
Core Philosophy
01

Depth before breadth

Each phase depends on the one before it. If something hasn't clicked, moving forward just adds confusion on top of confusion.

02

Every concept gets a project

Python is the main implementation language, but C comes in Phase 2 and stays. Reading source code without having written something similar is guessing.

03

The harder direction first

Most people learn frameworks and work downward. This goes the other way. Once you've built the lower layers, everything above them is just reading documentation.

0
Foundation
Regrounding — How Computers Work
4–6 weeks
▾
Goal

Reconstruct the mental model of how a computer actually works, end to end. This is the substrate everything else sits on. Without this, every upstream concept floats without an anchor.

Central Question
❓

What is actually happening when code runs?

Resources
📖
Code by Charles Petzold Reads like a story. Builds a computer from first principles — electricity → logic gates → CPU. Closes more gaps than a semester course. A cheap used copy is perfect.
🖥️
nand2tetris — From NAND to Tetris Free course. You build a complete computer from logic gates up to a running program. Do Part I (hardware, Chapters 1–6). Makes ARM/x86 make sense permanently.
↗ nand2tetris.org (free)
Project

Complete nand2tetris Part I (Chapters 1–6). By the end you will have built an ALU, CPU, and memory unit from scratch in a hardware simulator. This makes computer architecture permanently concrete.

1
Systems
Operating Systems
6–8 weeks
▾
Goal

Solidify processes, threads, memory, and system calls — topics you already showed genuine interest in. Your reading on threads and coroutines will make this feel like coming home. The concurrency section will finally make coroutines make complete mechanical sense.

Central Question
❓

How does the OS manage running programs, and how does my code interact with hardware through it?

Resources
📗
Operating Systems: Three Easy Pieces (OSTEP) Free online. Best modern OS textbook, written with a clarity the Dragon Book lacks. Covers virtualisation, concurrency, and persistence. Read it top to bottom.
↗ ostep.org (free)
Project

Write a small shell in Python or C: fork processes, handle signals, implement piping. Deceptively simple; teaches everything about how the OS actually works from the user's perspective.

2
Systems
C & Systems Programming
4–6 weeks
▾
Goal

You need C — not to master it, but to read it fluently. The Flutter engine, most runtimes, and every OS are written in C/C++. You cannot read source code in this domain without it.

Central Question
❓

What does memory actually look like, and what is a pointer?

Resources
📕
The C Programming Language — Kernighan & Ritchie (K&R) Short, dense, canonical. Written by C's creators. Read it cover to cover and do every exercise. This is the definitive source.
📘
CS:APP — Computer Systems: A Programmer's Perspective Chapters 1–3 for assembly and memory layout. Makes the gap between C and machine code completely visible.
↗ csapp.cs.cmu.edu
Project

Rewrite your Phase 1 shell in C, then write a memory allocator (malloc/free from scratch). This makes garbage collection and runtime memory management permanently clear — you'll never wonder how a GC works again.

3★
Core Domain — The Heart of This Roadmap
Compilers
10–14 weeks
▾
Goal

This is the core domain. By now your OS and architecture knowledge makes every step of compilation land properly. You will understand why each stage of the pipeline exists, not just what it does.

Central Question
❓

How does text become a running program?

Resources — do in this order
🥇
1. Crafting Interpreters — Robert Nystrom Start here. You build two complete interpreters for a real language (Lox): one tree-walking, one bytecode VM. The best compiler book written in the last 20 years. Free online.
↗ craftinginterpreters.com (free)
📙
2. Engineering a Compiler — Cooper & Torczon After Crafting Interpreters, for the formal/academic treatment: CFGs, register allocation, SSA form, optimisation passes. More approachable than the Dragon Book.
🐉
3. The Dragon Book — Aho, Lam, Sethi & Ullman Use as a reference, not a primary read. Encyclopedic, not pedagogical. It earns its place on your shelf after you've done the above two.
What You'll Understand After This Phase
Lexing, parsing, and AST construction
Scope, binding, and type checking
IR design and optimisation passes
Bytecode VMs and AOT compilation
Why Dart has both a VM (debug mode) and AOT (release mode) — completely transparent
Project

Finish both parts of Crafting Interpreters (the book guides you step by step). Then extend Lox with one feature of your own design — a new type, a new control structure, or a small standard library. This is where deep understanding is confirmed.

4
Theory
Programming Language Theory
8–10 weeks
▾
Goal

The theoretical underpinning of everything above: type systems, formal semantics, what "correct" means mathematically. This is where you stop taking language design decisions on faith.

Central Question
❓

Why are languages designed the way they are, and what guarantees can a type system actually give you?

Resources
📗
PLAI — Programming Languages: Application and Interpretation By Shriram Krishnamurthi. Free online. More accessible entry point. Read this before TAPL.
↗ plai.org (free)
📘
TAPL — Types and Programming Languages — Benjamin Pierce The canonical PL theory book. Dense but extraordinary. Go chapter by chapter; do not rush it.
Project

Implement a type checker for a small language in Python. Then add Hindley-Milner type inference — the algorithm that powers Haskell and OCaml's type systems. Genuinely difficult and genuinely rewarding.

5
Application
Runtimes & the Full Ecosystem
6–8 weeks
▾
Goal

Apply everything to the real world. You can now read actual runtime source code and understand it. The Flutter/Dart/iOS ecosystem that originally prompted this roadmap will now have zero gaps.

Central Question
❓

How does the Flutter/Dart/iOS ecosystem work at every layer, with no gaps?

Resources
🎯
Dart VM Source & Documentation Read with fresh eyes — it will now be readable. Study the VM's GC, isolates, and AOT pipeline.
↗ github.com/dart-lang/sdk
🦋
Inside Flutter — Flutter Engine Architecture Official deep-dive into how Flutter's rendering pipeline, text, and platform channels work internally.
↗ docs.flutter.dev/resources/inside-flutter
♻️
GC Algorithms Study mark-and-sweep and generational GC. Dart uses both. After Phase 3 these will make immediate sense.
🔗
FFI & Platform Channels How Dart calls C++ which calls the ObjC runtime. Where abstraction boundaries are drawn and why they exist there.
↗ dart.dev/interop/c-interop
Project

Write a garbage collector in Python for your Lox interpreter from Phase 3. Then write a small FFI bridge between Python and a C library. By this point you are building the kinds of things that underpin every language runtime.

Where to Begin

Start with just two things.

Don't buy everything at once. Begin here. If these click and excite you — and they will — the roadmap is confirmed. Return to this page as you reach each phase.
